# 创业红绿灯 - 项目策划书

> 给 Codex 使用。目标：先完成可演示、可路演、可继续迭代的黑客松 MVP。  
> 当前项目定位已确定，后续不要再发散成“创业成功率预测器”。

---

## 1. 项目名称

**创业红绿灯**

副标题：

> 根据真实验证证据，判断项目现在该推进、验证、暂停，还是该被行动倒逼。

---

## 2. 一句话定位

面向一人公司、独立开发者、AI 创作者的持续创业节奏校准 Agent。

它不预测项目成功率，也不是一次性咨询报告。它根据用户已经做过的真实验证行为，持续判断当前证据是否足够，并在想法、调研、Demo、MVP、运营各阶段给出下一步低成本行动。

每一轮校准都要保存；用户完成现实任务、回填新证据后重新判灯，形成“判断 → 行动 → 回填 → 再校准”的长期陪跑闭环。

---

## 3. 解决的问题

一人公司常见两种错误：

1. **冲动创业**：刚有想法就开发、花钱、外包、买域名、做完整产品。
2. **保守创业**：一直想、一直打磨、一直学习工具，但不找真实用户验证。

本项目核心价值：

> 让创业行动和证据匹配。证据不足时限制投入；长期不行动时倒逼真实验证。

---

## 4. 核心产品机制

```text
输入项目想法
↓
Agent 拆解关键假设
↓
用户填写已做过的验证行为
↓
计算：项目结构清晰度 + 证据充分度
↓
红队 Agent 只追问证据不足处
↓
输出：绿灯 / 黄灯 / 红灯 / 蓝灯
↓
生成下一步行动包
↓
用户回填真实结果并保存本轮校准
↓
根据新证据重新判灯，进入下一轮
```

---

## 5. 四种灯号

| 灯号 | 含义 | 判断 |
|---|---|---|
| 绿灯：推进 | 可以做 MVP 或继续迭代 | 有真实用户反馈、试用、付费或留存证据 |
| 黄灯：验证 | 方向可能成立，但证据不足 | 只允许做 3-7 天低成本验证 |
| 红灯：暂停 | 当前风险过高，不建议继续投入 | 用户、获客、付费、交付等关键假设薄弱 |
| 蓝灯：去验证 | 针对一直不行动的人 | 反复打磨方案，但没有任何外部反馈 |

注意：不要输出“成功率 XX%”。只输出“证据充分度”和“行动建议”。

---

## 6. 两套评分体系

### 6.1 项目结构清晰度（不是可行性预测）

判断项目逻辑是否成立：

- 用户是否明确
- 痛点是否真实
- 技术是否可做
- 获客是否可行
- 变现是否清楚
- 交付是否可控

其中“交付是否可控”必须给路径建议：

| 状态 | 建议 |
|---|---|
| 一人可做 | 直接做 MVP |
| 一人可做但要砍范围 | 缩小功能，只保留验证核心 |
| 一人不可做但可合作 | 建议找合作伙伴，并说明缺什么能力 |
| 一人不可做且合作成本高 | 暂停或换方向 |

### 6.2 证据充分度

判断用户已经做过哪些真实验证行为：

- 是否做过竞品/资料搜索
- 是否访谈过目标用户
- 是否发布过测试内容
- 是否有人留言、私信、加微信
- 是否有人试用 Demo
- 是否有人愿意付费
- 是否有复用、留存、转介绍

核心原则：

> 证据越充分，才允许投入更多资源。

---

## 7. 证据阶梯

| 等级 | 用户行为 | 建议 |
|---|---|---|
| 0 | 只有想法 | 不开发，先拆假设 |
| 1 | 做过资料/竞品搜索 | 继续调研 |
| 2 | 访谈过 3-5 个用户 | 做手动 Demo |
| 3 | 发过测试内容且有人反馈 | 做低成本 MVP |
| 4 | 有人试用原型 | 继续迭代 |
| 5 | 有人愿意付费/预订 | 可加大开发 |
| 6 | 有复用、留存、转介绍 | 可系统投入 |

## 8. P0 功能范围

P0 必须完成，作为黑客松可演示版本。

1. 首页
2. 项目输入页
3. 关键假设拆解页
4. 验证行为记录页
5. 证据充分度评分页
6. 红队质询页
7. 灯号判定页：绿灯 / 黄灯 / 红灯 / 蓝灯
8. 投入上限

10. 报告复制/导出 Markdown
11. 本轮校准保存与本地校准历程
12. 证据回填后重新校准，并说明下一次应复查的时机

## 9. P1 加分功能

P1 是加分项，不影响 P0 交付。

1. 案例库/方法论库
2. 行动包生成器
3. 合作伙伴建议
4. 用户访谈话术
5. 测试发布文案
6. 外包/请人需求说明
7. AI 工具执行提示词

行动包不绑定 Codex，也不假设用户会用 AI。输出四种版本：

- 手动执行版
- 合作伙伴版
- 外包/请人版
- AI 工具版

## 10. 红队 Agent 角色

| 角色 | 追问方向 |
|---|---|
| 用户 Agent | 目标用户为什么要用？不用你的产品时如何解决？ |
| 竞品 Agent | 已有替代方案是什么？你差异在哪里？ |
| 获客 Agent | 第一批用户从哪里来？不是泛泛说“发小红书”。 |
| 付费 Agent | 用户为什么愿意付钱？有没有付费信号？ |
| 执行 Agent | 当前资源能不能交付？一人做不了时找谁合作？ |
| 怀疑 Agent | 哪些只是创始人的幻想？哪些证据还不存在？ |

红队原则：

> 不鼓励用户盲目做，也不让用户一直空想。只根据证据推动下一步行动。

---

## 11. 案例库规划

案例库不是用来预测成功率，只用于：

- 提供方法论参考
- 提醒常见失败模式
- 辅助红队追问
- 标注证据缺口

黑客松版先做 30-50 条结构化卡片即可。

来源建议：

| 类型 | 来源 | 用途 |
|---|---|---|
| 方法论卡 | YC Startup Library / Startup School | 创业验证原则 |
| 成功案例卡 | Indie Hackers Stories | 成功项目如何获取第一批用户 |
| 失败案例卡 | Failory | 常见失败原因 |
| 失败模式卡 | CB Insights | 失败原因分类 |
| 产品参考卡 | GitHub idea validator 项目 | 参考竞品和功能边界 |
| 个人项目卡 | 用户自己的项目 | 自用案例、现场演示 |

P0 可先用本地 JSON；P1 再做向量检索或实时搜索。

## 12. 页面结构

### 12.1 首页

核心文案：

> 别盲目冲，也别一直想。先看看你的创业项目现在是什么灯。

按钮：

> 检查我的项目状态

### 12.2 项目输入页

字段：

- 项目名
- 一句话描述
- 目标用户
- 用户痛点
- 现有替代方案
- 获客方式
- 变现方式
- 当前阶段
- 已投入时间
- 已投入金钱
- 距上次真实用户行动的天数
- 最大不确定性

### 12.3 假设拆解页

输出：

- 用户假设
- 痛点假设
- 付费假设
- 获客假设
- 技术假设
- 交付假设

### 12.4 证据记录页

用勾选 + 数字输入：

- 访谈人数
- 测试内容发布次数
- 留言/私信人数
- 试用人数
- 付费/预订人数
- 复用/留存信号

### 12.5 红队质询页

展示 3-6 个 Agent 角色，根据证据缺口追问。

### 12.6 判定报告页

输出：

- 当前灯号
- 项目结构清晰度（不代表项目可行性）
- 证据充分度
- 最大风险
- 缺失证据
- 投入上限
- 7 天验证任务
- 停止条件
- 本轮重点
- 下次校准时机
- 校准历程

## 14. 技术栈建议

优先选择快速、稳定、可部署：

- Next.js / React
- TailwindCSS
- API Route 调用模型
- OpenAI Next API：`https://api.openai-next.com/v1`
- 本地 JSON / localStorage 起步
- 后续可接 Supabase / SQLite / 向量库

API Key 必须放环境变量，不要写死在前端。

---

## 15. 不做清单

P0 不做：

- 登录系统
- 复杂数据库
- 全网实时爬虫
- 自动预测成功率
- 支付系统
- 自动发小红书
- 自动调用 Codex
- 大而全项目管理系统

---

## 16. 黑客松演示案例

优先准备 4 个案例：

1. AI 试衣 Agent：技术可行，商业证据不足，黄灯验证。
2. 电商退货归因 Agent：商业价值强，但需要行业数据/队友，黄灯或合作建议。
3. 今天也要守住小游戏：已有 Demo，需要判断继续迭代还是做传播测试。
4. 创业红绿灯自身：作为自举案例，展示从想法到证据驱动开发。

## 17. 路演 PPT 后续任务

项目完成后必须制作路演 PPT。

参考用户提供的路演 PDF 的结构，不照搬视觉，但参考叙事顺序：

1. 封面：一句话定位
2. 亮点：3-4 个核心亮点
3. 项目简介
4. 场景痛点
5. 产品流程/架构
6. 核心机制：证据充分度 + 红绿灯判定
7. Demo 展示
8. 落地场景/应用案例
9. 后续规划
10. 结束页：价值主张

PPT 风格后续再升级，建议方向：

- 深色科技感 + 红黄绿信号灯视觉
- 证据卡片 / 风险仪表盘 / 行动任务卡
- 少字、大标题、强对比
- 每页只讲一个结论

---

## 18. Codex 开发要求

1. 优先完成 P0，不要发散。
2. 所有页面中文界面。
3. 不频繁要求用户测试，Codex 先自检。
4. 只有关键节点才请用户验收。
5. 每次输出：修改文件列表、实现说明、自测结果、待确认问题。
6. 不硬编码 API Key。
7. 模型调用失败时必须有 fallback 示例数据，保证演示不崩。
8. UI 要有“红绿灯”主题，不要做成普通聊天框。
