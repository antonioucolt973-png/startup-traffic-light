# OPC 创业红绿灯｜阶段 2 比赛演示说明

## 一、演示目标

本阶段要让评委看到一条完整闭环：

```text
一句模糊创业想法
→ 三轮 AI 澄清
→ 可复核的分析摘要
→ 选择商业路径
→ 用户在 AI 引导下写出自己的方案
→ 红队压力测试与风险修改
→ 生成商业化落地地图
→ 转换成可执行任务
```

核心表达：OPC 不是替创业者写答案的普通聊天工具，而是一个把创业想法转换为“待验证假设、风险和现实任务”的 AI 验证陪跑系统。

本版统一使用“AI 一键换衣助手”作为比赛案例。比赛现场未连接外部大模型和实时搜索，AI 澄清、资料分析和红队测试由本地预设稳定生成；后续商业化可以在相同数据结构上接入大模型和联网搜索。

## 二、建议演示时间

- 标准演示：约 5 分钟，完整走完主要流程。
- 压缩演示：约 2 分钟，只展示关键页面和代表性操作。
- 不建议在 2 分钟内逐字介绍全部风险和所有字段，会导致评委抓不住重点。

## 三、演示前准备

1. 启动项目：

   ```bash
   npm run dev
   ```

2. 打开终端显示的本地地址。
3. 停留在左侧导航“项目出发”。
4. 建议浏览器缩放为 90%～100%，分辨率使用 1440 × 900 或更高。
5. 如果浏览器保留了上一次项目状态，回到“项目出发”重新输入案例。
6. 正式演示前完整走一遍，确保熟悉六阶段路口决策按钮的位置。

左侧创业流程应显示：

1. 项目出发
2. 路线总览
3. 路口决策
4. 任务执行
5. 证据背包
6. 成长回顾

## 四、标准演示脚本

### 第一步：项目出发——把模糊想法整理成一句话

#### 操作

1. 点击“不会描述？用引导描述”。
2. 填写模板：
   - 我想帮：`经常网购服装的年轻女性`
   - 解决：`下单前不知道衣服上身效果是否适合自己`
   - 通过：`AI 一键换衣`
   - 让他们可以：`先看到自己的穿衣效果再决定是否购买`
   - 我目前有的资源：`一个可点击的换衣 Demo 和 5 位愿意试用的朋友`
3. 点击“一键导入到上方聊天框”。
4. 检查完整内容已经进入上方“我想做一个”输入框。
5. 点击“让 AI 帮我理清”。

#### 现场讲解

> 用户不需要会写商业计划书。填空模板只负责整理输入，并不会直接替用户作出判断；导入聊天框后，用户仍然可以修改。

### 第二步：完成三轮 AI 澄清

每轮只回答一个会影响验证路线的问题。

#### 操作

1. 第一轮选择：`经常网购服装、担心上身效果的年轻女性`。
2. 点击“确认并进入下一问”。
3. 第二轮选择：`下单前确认衣服是否适合自己`。
4. 点击“确认并进入下一问”。
5. 第三轮选择：`可点击的换衣 Demo 或效果图`。
6. 点击“完成澄清，进入路线总览”。

#### 现场讲解

> 三轮澄清分别确认第一批用户、真正要解决的问题，以及现在能够用于最低成本验证的资源。系统不会无限聊天，也不会一上来要求开发完整产品。

### 第三步：路线总览——查看分析摘要并选择商业路径

进入左侧第二项“路线总览”后，先介绍上半部分 AI 分析工作台，再介绍下半部分商业路径。

#### 1. 展示 AI 分析工作台

按以下顺序讲解：

1. “AI 分析过程”：默认展示问题拆解、市场与替代扫描、商业路径推演。
2. 点击“展开完整 5 步分析与详细摘要”，查看关键假设和最小验证路线。
3. “AI 已识别”：展示目标用户和核心问题。
4. “AI 当前判断”：区分用户明确输入和仍待验证的假设。
5. “AI 使用的方法”：展示用户问题拆解、假设树和最小成本验证方法。

#### 必须说明

> 页面展示的是可检查的结构化分析摘要，不展示模型内部逐字推理。比赛版结果由本地规则和案例库生成，也不会被当成真实用户证据。

#### 2. 选择商业路径

1. 依次点击路径 A、路径 B、路径 C，让评委看到每条路径具有统一结构。
2. 每条路径包括：
   - 路径定位和适用条件；
   - 市场验证假设；
   - 商业模式；
   - 关键成功因素；
   - 周期、难度和比赛预设成功概率；
   - 选择、查看详细拆解和修改操作。
3. 演示推荐选择路径 A“轻量级 SaaS 工具路线”。
4. 点击“选择这条路径”。
5. 点击“确认路线并进入路口决策”。

#### 现场讲解

> 这里生成的是三条待验证的商业假设，不是市场结论。创业者先选择与当前资源最匹配的一条，再进入路口决策把它写成自己的方案。

### 第四步：路口决策——使用顶部六阶段导览

进入左侧第三项“路口决策”后，首先指向页面上方的阶段导览：

```text
方案引导 → 方案确认 → 资料分析 → 红队测试 → 测试结论 → 路线地图
```

#### 现场讲解

> 顶部导览显示整个决策进度。绿色表示已经完成，当前阶段会高亮，尚未完成前置步骤的阶段会锁定；完成后可以返回前面的阶段修改。

### 第五步：阶段一“方案引导”——五步写出自己的方案

阶段内部还有五个步骤：

```text
Who → What → Why → How → When & Where
```

比赛演示时，为了提高速度，每一步都使用预设完善功能。

#### 操作

1. Who“明确目标用户”：
   - 点击“让AI帮我完善”；
   - 简单展示人口特征、心理特征、行为特征和用户画像；
   - 点击“保存并进入下一步”。
2. What“定义核心问题”：
   - 点击“让AI帮我完善”；
   - 点击“保存并进入下一步”。
3. Why“提出价值主张”：
   - 点击“让AI帮我完善”；
   - 点击“保存并进入下一步”。
4. How“设计商业模式”：
   - 点击“让AI帮我完善”；
   - 点击“保存并进入下一步”。
5. When & Where“规划执行路径”：
   - 点击“让AI帮我完善”；
   - 点击“保存这一步”。
6. 点击页面下方“完成方案框架”。

#### 现场讲解

> AI 不直接替用户决定最终方案，而是依次追问 Who、What、Why、How 和 When。用户可以采用 AI 的参考内容，也可以修改或自己填写。

### 第六步：阶段二“方案确认”

#### 操作

1. 展示五个方案模块的汇总结果。
2. 指出每个模块都有明确对应的“修改目标用户”“修改核心问题”等入口。
3. 点击“确认方案并开始分析”。

#### 现场讲解

> 红队测试不会针对一段模糊聊天内容，而是针对用户主动确认后的结构化方案版本。

### 第七步：阶段三“资料分析”

#### 操作

1. 指出页面的比赛预设提示。
2. 点击“开始资料分析”。
3. 展示五类资料：
   - 市场规模；
   - 竞品情况；
   - 行业趋势；
   - 用户评论；
   - 政策法规。
4. 点击“确认资料并启动红队测试”。

#### 必须说明

> 当前没有伪装成实时联网。比赛版使用本地案例库保证演示稳定，商业化版本可以在这里连接真实搜索和大模型，并保留相同的资料结构。

### 第八步：阶段四“红队测试”

系统会依次检查6个风险：

1. 换衣效果准确度不足；
2. 用户不愿上传个人照片；
3. 个人用户付费意愿低；
4. 获客成本可能高于收入；
5. 电商平台可能快速复制功能；
6. 肖像、隐私和图片存储风险。

每个风险卡包括发现的问题、分析依据、风险影响以及3个缓解方案。

#### 推荐演示操作

为了快速完成流程，依次选择：

1. 准确度风险：选择方案1“先验证决策帮助度”。
2. 照片上传风险：选择方案1“先测上传意愿”。
3. 付费风险：选择方案1“先做免费MVP验证付费意愿”。
4. 获客风险：选择方案1“先测试私域转介绍”。
5. 平台竞争风险：选择方案3“先做店铺服务”。
6. 隐私风险：选择方案1“测试阶段不长期存图”。
7. 点击“完成压力测试”。

#### 可选演示

- 点击“我想自己写一个方案”，展示用户可以拒绝 AI 建议并自己填写。
- 点击“这个风险我接受”，展示系统允许创业者主动承担风险，但会记录在测试结论中。
- 点击风险进度圆点，查看已经处理过的风险。

#### 现场讲解

> 红队不是只说“有风险”，而是要求用户对每个风险选择缓解方案、自定义方案或主动接受。每次选择都会形成方案更新记录。

### 第九步：阶段五“测试结论”

#### 操作

1. 展示测试风险数、缓解方案数、接受风险数和测试轮数。
2. 展示红队建议：`先小范围验证`。
3. 展示“方案更新记录”，说明前面选择的缓解方案已经写入结果。
4. 勾选“我确认压力测试已完成”。
5. 点击“确认并绘制路线地图”。

#### 现场讲解

> 系统不会因为用户有一个 Demo 就建议扩大开发。当前结论是可以小范围验证，但照片上传意愿、换衣可信度和付费行为仍需要真实证据。

### 第十步：阶段六“路线地图”

#### 操作

展示4个商业化里程碑：

1. M1 验证需求；
2. M2 优化换衣 Demo；
3. M3 获取种子用户；
4. M4 验证付费或店铺合作。

每个里程碑都包含：

- 阶段目标；
- 预计周期；
- 具体任务；
- 成功标准；
- 主要风险；
- 停止条件。

最后点击“立即出发，进入任务执行”。

#### 现场讲解

> 这张地图不是一份宏观商业计划书。它把经过风险测试的方案拆成能执行、能判断成功、也能及时停止的现实任务。

### 第十一步：任务执行

进入左侧第四项“任务执行”后，路线地图已经自动转换成按里程碑分组的现实任务。

#### 1. 展示任务导览

页面上方按4个里程碑分组：

1. M1 验证需求；
2. M2 优化换衣Demo；
3. M3 获取种子用户；
4. M4 验证付费或店铺合作。

当前任务可以进入，尚未解锁的任务显示锁定状态。证据通过后才会解锁下一项任务。

#### 2. 展示任务卡

按照图中顺序介绍任务卡：

1. 任务编号和任务名称；
2. 目标；
3. 具体动作；
4. 执行工具；
5. 预计耗时；
6. 预计成本；
7. 成功标准；
8. 提交证据；
9. 标记完成、遇到困难、延期执行、跳过这步。

点击任意一个执行工具，例如“用户访谈提纲”，展示预设模板，再点击“复制模板”。

#### 3. 演示一次证据充分

在第一个任务中填写：

- 实际发生了什么：`访谈5位经常网购服装的年轻女性，其中4人说下单前会反复查看买家秀，3人愿意上传照片体验换衣Demo。`
- 涉及人数或次数：`5`
- 用户原话：`如果能少退货，我愿意先上传照片试一次。`
- 证明材料：选择“粘贴数据”，填写：`5人访谈：4人承认问题，3人愿意上传照片，1人明确拒绝。`

点击“标记完成”。

预期结果：

- 显示“证据充分”；
- 当前任务标记为“证据已通过”；
- 下一项任务解锁；
- 证据自动进入证据背包；
- 项目灯号不会因为任务文字自动变化，只会读取已确认的现实证据。

#### 4. 可选演示一次证据不足

如果要展示补证机制，在另一任务中只填写简短描述、人数1，不填写用户原话或证明材料，然后点击“标记完成”。

预期显示：

> 证据收到，但似乎还需要更多数据支撑。

可选择：

- 补充证据；
- 先继续；
- 求助。

#### 5. 可选展示其他任务操作

- “遇到困难”：填写联系不到目标用户，查看比赛预设替代方案；
- “延期执行”：填写延期时间和原因；
- “跳过这步”：记录跳过原因并保留状态。

#### 现场讲解

> 用户不能只靠点击“完成”让项目前进。任务必须产生现实行为记录，再经过证据质量检查；证据不足时，系统会要求补证或明确记录风险。

### 第十二步：证据背包

点击“查看证据背包”或左侧第五项“证据背包”。

#### 操作

1. 展示顶部统计：证据总数、已确认、需要补证、已排除。
2. 指出证据按照 M1～M4 里程碑分组。
3. 展开刚才提交的证据。
4. 展示：
   - 对应任务；
   - 样本数量；
   - 提交时间；
   - 证明材料；
   - 实际发生的行为；
   - 用户原话；
   - 数据记录；
   - AI证据质量评估。
5. 对于“需要补证”的记录，点击“补充证据”会返回对应任务。
6. 点击“修改说明”只能修改记录内容，不能手动把证据改成已确认。
7. 点击“排除无效证据”后，该记录不再计入证据充分度。

#### 现场讲解

> 证据背包不是随意上传材料的网盘，而是任务结果的可追溯账本。每条证据都能返回产生它的任务，只有审核确认后的现实行为才会影响项目灯号。

### 第十三步：成长回顾

点击左侧第六项“成长回顾”。本页不再执行任务或编辑证据，只负责结算当前轮次和启动下一轮。

#### 1. 展示本轮结算

1. 查看“当前结论”，例如：`保持当前阶段，继续小范围验证`。
2. 展示“这一轮已经得到”和“这一轮还没有证明”。
3. 展示已执行任务、已确认证据、需要补证和当前灯号。
4. 查看证据充分度从本轮起点到当前的变化。
5. 点击“查看证据依据”可以返回证据背包。

#### 2. 展示假设结算

系统会逐项结算：

- 目标用户是否真实存在问题；
- 用户是否愿意上传照片；
- 换衣结果是否会改变购买判断；
- 是否能低成本获得种子用户；
- 个人用户或服装店是否愿意付费。

结论只使用：

- 已支持；
- 初步支持；
- 证据不足；
- 尚未验证。

点击“查看证据”可以检查结论对应的现实记录。

#### 3. 展示本轮成长变化

按“起点 → 现在”介绍：

- 用户理解；
- 核心问题；
- 商业路径；
- 验证能力。

#### 4. 展示AI阶段复盘

页面固定展示：

- 本轮得到；
- 下降的风险；
- 当前最高风险；
- 下一轮唯一目标。

必须向评委说明：AI复盘只读取已确认证据，待补证和已排除证据不会支持结论。

#### 5. 选择下一轮方向

三个选项：

1. 继续前进：证据达到升级门槛时才可选择；
2. 保持阶段：证据不足但方向仍值得验证，当前案例推荐此项；
3. 返回修正：核心假设被否定时退回上一阶段。

修改或确认“下一轮唯一目标”，然后点击“开启第2轮”。

新一轮会继承项目资料、历史证据、风险记录和任务结果；本轮数据会归档，不会被覆盖。

#### 现场讲解

> 成长回顾不是庆祝完成了多少任务，而是结算哪些假设被现实支持、哪些仍然未知。下一轮能否升级由证据门槛决定，用户不能绕过规则强行前进。

至此结束标准演示。

## 五、2分钟压缩版演示

时间紧张时按以下节奏：

### 0:00—0:25 项目出发

- 使用填空模板导入一键换衣想法。
- 快速完成三轮预设澄清。
- 强调每轮只确认一个影响验证路线的问题。

### 0:25—0:50 路线总览

- 展示可复核分析摘要。
- 强调不展示内部逐字推理、不把AI结论当证据。
- 选择路径A，进入路口决策。

### 0:50—1:20 路口决策

- 指出顶部六阶段导览。
- 在“方案引导”只演示一次“让AI帮我完善”。
- 口头说明其余 Who/What/Why/How/When 使用同一方式完成。
- 如果现场已经提前准备好完整状态，可以直接展示红队测试页面。

### 1:20—1:45 红队测试与结论

- 展示一个高危风险及3个缓解方案。

## 六、评委可能追问及回答

| 问题 | 建议回答 |
| --- | --- |
| 没有大模型还能演示吗？ | 可以。比赛版使用本地预设分析和风险案例，保证现场稳定；商业化版本会在相同结构处连接大模型和实时搜索。 |
| 这是不是普通聊天机器人？ | 不是。系统限制每一阶段的输入和输出，把对话转化为用户、问题、价值、商业模式、风险、里程碑和任务。 |
| 为什么不直接让AI生成完整方案？ | 直接生成会让用户失去判断过程。这里由AI提问和提供参考，最终方案、风险选择和接受结果都由用户确认。 |
| 页面为什么不展示AI完整思考过程？ | 产品展示可复核的分析步骤、中间产出和依据摘要，不展示模型内部逐字推理。 |
| 资料分析是真实联网结果吗？ | 比赛版不是，页面已明确标注为本地案例库预设。商业化版本才接实时搜索，并显示来源和时间。 |
| AI输出会影响红绿灯吗？ | 不会。AI输出属于假设和建议，只有任务执行后产生的可复核用户行为证据才会进入证据背包并影响灯号。 |
| 为什么任务执行在证据背包前面？ | 因为先完成访谈、试用、报价等现实任务，才会产生能够放进证据背包的材料。 |
| 为什么不先开发完整换衣产品？ | 当前最大不确定性是用户是否愿意上传照片、信任效果并采取付费行动，而不是能否继续增加功能。 |
| 风险测试是不是只给负面评价？ | 不是。每个风险都必须提供可选择的缓解方案、成本、周期和验证方式，用户也可以自己提出方案或主动接受风险。 |
| 点击“标记完成”就算任务通过吗？ | 不算。系统会先检查对象、真实行为、数量、用户原话和证明材料；证据充分才会通过并解锁下一任务。 |
| 比赛版真的会上传截图或录音吗？ | 比赛版在本地记录文件名称和证据说明，不向外部服务器上传。商业化版本再连接对象存储和权限控制。 |
| 用户能自己把证据改成已确认吗？ | 不能。用户可以修改说明、补充材料或排除记录，确认状态来自证据审核流程。 |
| 用户能强制选择“继续前进”吗？ | 不能。只有灯号和关键证据达到规则门槛时才会解锁前进；否则只能保持阶段或返回修正。 |
| 成长回顾中的AI结论依据是什么？ | 只读取已确认证据和任务结果，待补证、已排除记录以及AI自己的建议都不能支持假设。 |

## 七、演示时不要说错的内容

1. 不要说“AI已经证明市场存在”。只能说“AI提出了待验证的市场假设”。
2. 不要说“正在联网搜索真实数据”。比赛版是本地预设资料分析。
3. 不要说“成功概率是科学预测”。页面数字是比赛预设，用于展示数据结构。
4. 不要把朋友愿意试用说成已经完成真实试用。
5. 不要把生成的换衣效果说成已经达到商业级准确度。
6. 不要说AI决定了最终方案。最终方案和风险承担由用户确认。
7. 不要把“标记完成”说成任务已经通过，必须说明还要进行证据评估。
8. 不要声称比赛版文件已经上传云端；当前只在本地记录文件名称。
9. 不要把成长积分或项目车前进当成市场验证结论。
10. 不要说用户可以任意升级阶段；“继续前进”受证据门槛限制。

## 九、推荐收尾话术

> 很多创业项目失败，不是因为团队不会开发，而是因为没有先验证最关键的假设。OPC 创业红绿灯让 AI 帮助创业者拆解问题、接受红队压力测试，并把方案转换成现实任务；最终决定项目继续、调整还是停止的，不是 AI 的想象，而是真实用户行为和证据。
