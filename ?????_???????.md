# 创业红绿灯：项目与技能推荐整理

## 1. 项目核心目标

创业红绿灯的目标不是预测创业项目一定成功，也不是生成一份看起来完整的商业计划书。

它真正要解决的问题是：

> 帮助一人公司、独立开发者和副业创业者判断：一个想法是否值得继续投入时间、金钱和开发资源。

因此，核心能力应围绕以下六点构建：

1. 需求验证：是否存在真实用户和真实痛点。
2. 市场研究：是否有类似成功案例、失败案例和可参考市场。
3. 竞品拆解：现有产品如何获客、收费和交付。
4. 实验设计：下一步如何用最低成本验证。
5. 证据管理：避免 AI 编造数据、市场规模和用户需求。
6. 报告生成：让结论可用于决策、路演和团队讨论。

---

## 2. 关键判断

创业红绿灯不应该做成：

```text
ChatGPT 套壳创业分析工具
```

因为这种产品的竞争力很低，任何通用模型都能生成类似内容。

更有差异化的方向是：

> AI 不直接告诉用户“项目能成功”，而是要求用户拿出真实证据，证明这个项目值得继续投入。

项目不应重点堆叠 Agent 数量，而应重点提升：

- 证据质量
- 判断门槛
- 失败条件
- 验证任务
- 结论可解释性

# 3. 推荐项目和技能

## 3.1 Evidence.dev

**平台：** GitHub / Web

**用途：**

- 数据分析报告生成
- 使用 SQL 与 Markdown 创建交互式报告
- 将结构化数据转成类似咨询报告的网页

**适合创业红绿灯的原因：**

创业红绿灯后期需要把访谈、用户反馈、竞品价格和验证数据整理成专业报告。

例如：

```text
项目：AI 宠物陪伴机器人

输入：
- 市场规模
- 竞品数量
- 用户访谈
- 搜索趋势
- 付费案例

输出：
- 需求强度图
- 痛点排名
- 验证进度
- 红黄绿灯结论
```

相比普通 AI 生成 PDF，Evidence.dev 更适合做结构化、可复用、可交互的分析报告。

**成本与风险：**

- 技术门槛中等
- 不适合最早期 Demo
- 需要先有结构化数据

**最小验证方式：**

制作一个简单报告页：

```text
输入：10 条用户访谈

输出：
- 用户痛点排名
- 需求强度
- 证据完整度
- 下一步验证建议
```

**优先级：S**

---

## 3.2 Lean Canvas Skill / Lean Canvas Agent

**平台：** GitHub / Agent Skill

**用途：**

自动生成：

- 用户群体
- 核心痛点
- 解决方案
- 渠道
- 收入模式
- 成本结构
- 核心指标
- 竞争优势

**适合创业红绿灯的原因：**

Lean Canvas 可以作为用户想法与红绿灯判断之间的结构化中间层。

建议流程：

```text
用户输入创业想法

↓

生成 Lean Canvas

↓

检查每一项是否有真实证据

↓

进行红黄绿灯判断

↓

生成 7 天验证计划
```

**关键限制：**

Lean Canvas 不能只是模板填充。

每个字段必须区分：

- 已确认事实
- 用户假设
- AI 推测
- 尚未验证内容

**成本与风险：**

- 接入成本低
- 容易退化为普通商业计划书生成器
- 必须绑定证据，否则价值有限

**最小验证方式：**

输入一个创业想法，生成 Lean Canvas，并要求每一栏同时输出：

```text
当前内容
证据来源
证据强度
待验证问题
```

**优先级：S**

---

## 3.3 Perplexica

**平台：** GitHub / 开源 AI 搜索

**用途：**

类似 Perplexity 的开源 AI 搜索引擎，可用于：

- 竞品搜索
- 市场案例搜索
- 产品价格调查
- 用户评价收集
- 失败案例查找
- 公开资料引用

**适合创业红绿灯的原因：**

创业红绿灯需要自动收集外部市场证据，而不能只依赖模型记忆。

例如用户输入：

```text
做 AI 健身教练是否有机会？
```

系统可以收集：

- 已融资公司
- 同类产品
- 产品价格
- 用户差评
- Reddit 或论坛讨论
- 失败案例
- 差异化方向

**成本与风险：**

- 需要部署和搜索配置
- 搜索结果质量不稳定
- 仍需做来源可信度判断
- 不能直接把搜索摘要当事实

**最小验证方式：**

输入：

```text
AI 宠物健康助手
```

输出：

- 10 个竞品
- 价格
- 核心功能
- 用户评价
- 差异点
- 资料来源

**优先级：S**

---

## 3.4 Browser-use

**平台：** GitHub / Browser Agent

**用途：**

让 Agent 自动操作浏览器，完成：

- 搜索竞品
- 访问产品网站
- 提取价格
- 查找用户评论
- 收集产品功能
- 整理公开市场信息

**适合创业红绿灯的原因：**

创业红绿灯未来的重要壁垒之一不是模型，而是：

> 自动帮助用户收集真实市场证据。

例如用户输入：

```text
我要做 AI 简历优化工具
```

Agent 可以自动：

1. 搜索同类产品。
2. 打开竞品网站。
3. 记录价格。
4. 提取核心卖点。
5. 查找用户差评。
6. 输出竞品表。

**成本与风险：**

- 技术复杂度较高
- 网页结构变化会导致失效
- 维护成本高
- 登录、验证码和反爬可能导致失败

**最小验证方式：**

只做一个低复杂度任务：

```text
输入一个创业项目名称
自动收集 5 个竞品
输出名称、网址、价格、核心功能
```

**优先级：A**

---

## 3.5 LangGraph

**平台：** GitHub / Agent 工作流框架

**用途：**

构建复杂、有状态、多步骤的 Agent 工作流。

适合的未来流程：

```text
用户想法

↓

市场研究 Agent

↓

竞品分析 Agent

↓

风险分析 Agent

↓

红队质疑 Agent

↓

验证计划 Agent

↓

报告 Agent
```

**适合创业红绿灯的原因：**

创业红绿灯本质上不是单轮聊天，而是一个多步骤判断流程，需要保存状态、证据和中间结果。

**成本与风险：**

- 架构复杂度中等
- 现在过早接入会增加开发成本
- 容易为了多 Agent 而多 Agent
- 不适合当前 P0 Demo

**建议：**

当前先使用简单规则引擎和单流程。

等以下条件满足后再考虑 LangGraph：

- 判断流程已经稳定
- 数据结构已经明确
- 不同分析步骤确实需要拆分
- 需要中断、恢复或人工介入

**优先级：A，后期使用**

---

## 3.6 Ragas

**平台：** GitHub / AI 评测框架

**用途：**

评估 RAG 与 AI 输出质量，例如：

- 是否有引用
- 是否忠于资料
- 是否回答问题
- 是否存在幻觉
- 结论与证据是否一致

**适合创业红绿灯的原因：**

创业红绿灯最大的风险之一，是 AI 一本正经地编造：

```text
该市场预计 2030 年达到 500 亿美元
```

但实际没有可靠来源。

Ragas 可用于后期建立固定测试集，检查：

- 是否引用真实资料
- 缺少证据时是否降低结论
- 是否把假设当成事实
- 是否编造市场数据

**成本与风险：**

- 偏技术
- 需要先建立高质量样本
- 没有明确评分标准时，评测结果价值有限

**最小验证方式：**

准备 20 个创业案例，对每个案例检查：

```text
1. 结论是否有证据支持
2. 无证据时是否明确标注
3. 是否出现编造数据
4. 同类输入是否输出稳定
```

**优先级：A**

---

## 3.7 Dify

**平台：** 开源 AI Agent 平台

**用途：**

快速搭建：

- 工作流
- 知识库
- Prompt 管理
- 模型调用
- API 接口
- 简单 Agent

**适合创业红绿灯的原因：**

适合快速做比赛 Demo，快速验证流程。

可用于：

```text
输入创业想法

↓

执行多个分析节点

↓

生成红黄绿灯结论

↓

输出验证计划
```

**个人判断：**

Dify 适合快速验证，不适合成为最终核心架构。

原因是：

创业红绿灯的竞争力不在工作流界面，而在：

- 判断规则
- 证据门槛
- 数据结构
- 验证逻辑
- 长期案例库

**成本与风险：**

- 上手快
- 容易形成平台依赖
- 自定义复杂判断时不够灵活

**优先级：A，适合 Demo**

---

## 3.8 GrowthBook

**平台：** GitHub / A/B 测试

**用途：**

测试不同产品方案的用户效果，例如：

- 红黄绿三色 vs 0–100 分
- 简版报告 vs 长报告
- 直接结论 vs 先展示证据
- 不同验证任务结构

**适合创业红绿灯的原因：**

产品上线后可以验证用户更容易理解和信任哪种表达方式。

**成本与风险：**

- 当前阶段不是核心
- 需要有真实流量才有意义

**优先级：B**

---

## 3.9 PostHog

**平台：** GitHub / 产品分析

**用途：**

分析用户行为，例如：

- 用户输入了什么项目
- 用户在哪一步退出
- 哪个报告被复制
- 哪类建议最常被点击
- 用户是否完成验证任务

**适合创业红绿灯的原因：**

上线后不能只看访问量，需要看用户是否真正完成：

```text
输入想法
→ 阅读分析
→ 接受验证计划
→ 执行验证
→ 回填证据
```

**成本与风险：**

- 功能较多
- 早期可能过重
- 需要做好隐私和数据处理

**优先级：B**

# 4. 推荐技术路线

## 4.1 V1：比赛版 / Demo 版

### 目标

先证明创业红绿灯的判断流程有效，不追求复杂多 Agent。

### 技术建议

```text
React + Vite + TypeScript
+
本地规则引擎
+
简单 LLM 分析和润色
```

### 核心流程

```text
用户输入创业想法

↓

项目分类

↓

证据检查

↓

红队质疑

↓

红黄绿灯判断

↓

7 天验证计划

↓

生成报告
```

### V1 必须包含

1. 用户输入创业想法。
2. 用户补充目标用户、预算、时间和已有证据。
3. 系统区分事实、假设和推测。
4. 系统给出红黄绿灯结论。
5. 系统解释为什么。
6. 系统生成低成本验证任务。
7. 系统明确什么情况下应该停止。

### V1 不建议加入

- 复杂多 Agent
- 自动浏览全网
- LangGraph
- 大型知识库
- 自动生成商业计划书
- 过多图表
- 复杂账户和权限体系

---

## 4.2 V2：产品版

在 V1 流程验证后，再增加：

```text
Perplexica
+
Browser-use
+
Lean Canvas
+
Ragas
```

### 新增能力

- 自动搜索竞品
- 自动提取价格
- 自动收集公开用户评价
- 自动生成证据引用
- 对输出进行幻觉检查
- 对创业假设进行结构化管理

---

## 4.3 V3：壁垒版

长期真正有价值的不是 Agent 数量，而是数据积累。

建议建立：

```text
创业案例数据库

+

真实验证结果

+

失败案例数据库

+

行业规则

+

用户执行结果
```

每个项目记录：

- 初始想法
- 初始评分
- 已有证据
- 系统建议
- 用户执行的验证任务
- 最终结果
- 是否继续开发
- 是否产生收入
- 失败原因

长期可形成：

- 不同行业的证据标准
- 不同类型项目的失败模式
- 不同投入规模下的成功概率参考
- 更可信的红黄绿灯规则

# 5. 建议优先级

## 第一阶段：必须吸收

1. Lean Canvas 的结构化分析方式。
2. 证据门槛机制。
3. 事实、假设、推测分类。
4. 红队质疑流程。
5. 7 天低成本验证任务。
6. 明确停止条件。

## 第二阶段：增强市场调查

1. Perplexica。
2. Browser-use。
3. 自动竞品收集。
4. 自动价格与用户评论整理。
5. 公开信息来源引用。

## 第三阶段：质量控制

1. Ragas。
2. 固定创业案例测试集。
3. AI 幻觉检测。
4. 结论稳定性测试。
5. 评分规则回归测试。

## 第四阶段：产品增长

1. PostHog。
2. GrowthBook。
3. 用户验证任务完成率。
4. 报告转化与留存分析。

# 6. 给 Codex 的实现建议

## 当前阶段建议

先不要重构成多 Agent，也不要马上引入 LangGraph。

优先检查当前 Demo 是否已经具备：

```text
1. 输入结构化
2. 证据强度判断
3. 红黄绿灯规则
4. 红队质疑
5. 验证任务
6. 停止条件
7. 报告输出
```

## 建议 Codex 先做的任务

```text
目标：
评估当前“创业红绿灯”Demo 与本方案的差距，并给出最小改进计划。

重点检查：
1. 是否区分事实、假设和推测。
2. 是否要求用户提供真实证据。
3. 是否存在明确红黄绿灯规则。
4. 是否能解释评分和结论。
5. 是否提供低成本验证任务。
6. 是否有停止条件。
7. 是否只是普通商业计划书生成器。
8. 是否需要引入 Lean Canvas 作为中间结构。
9. 是否适合后续接入 Perplexica 或 Browser-use。
10. 当前是否有必要使用 LangGraph。

限制：
- 不直接重构项目。
- 不安装新框架。
- 先只读分析现有代码和文档。
- 输出差距、优先级、最小修改范围和验收标准。
```

# 7. 明确结论

创业红绿灯当前最应该做的，不是增加更多 Agent，而是强化以下三点：

1. **证据门槛**
2. **低成本验证任务**
3. **可解释的继续或停止判断**

推荐顺序：

1. Lean Canvas
2. Perplexica
3. Browser-use
4. Ragas
5. LangGraph
6. Dify
7. PostHog / GrowthBook

最核心产品定位：

> 创业红绿灯不是帮助用户把想法说得更好，而是帮助用户更早发现：这个项目是否值得继续投入。

## 8. P0 实施后的外部 Skill 复核

### `ferdinandobons/startup-skill`

仓库定位是创业验证、竞品情报、定位与路演的多 Skill 工具集；它包含 `startup-design`、`startup-competitors`、`startup-positioning`、`startup-pitch` 等独立流程，并以 MIT 协议发布。

**本项目吸收：**

1. 用户访谈与真实行为优先于创始人自述。
2. 假设、证据、验证任务和停止条件必须分开呈现。
3. 对没有外部行为的项目，先推动一次真实触达，而不是生成更长的分析报告。

**本项目不直接接入：**

1. 不把整个 Skill 安装进 P0，不引入多 Agent、联网竞品搜索或长篇商业计划生成。
2. 不把竞品、定位、路演模块伪装成已发生的用户证据。
3. 不在证据不足时输出市场规模、成功率或确定性结论。

**结论：**

它对创业红绿灯有参考价值，但当前最正确的用法是提取“真实证据门槛、诚实输出、行动验证”三条原则。P0 已通过三段六关、蓝灯停滞触发、红队应对与校准历史实现这些原则；完整接入应留在 P1 之后再评估。
