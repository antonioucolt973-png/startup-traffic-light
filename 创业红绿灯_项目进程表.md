## 3. P0 详细任务

### P0-1：项目骨架与 UI

| 任务 | 说明 | 验收 |
|---|---|---|
| 建立项目页面结构 | 首页、项目输入、证据记录、红队质询、报告页 | 页面可访问，无报错 |
| 设计红绿灯主题 UI | 红/黄/绿/蓝灯状态、仪表盘、卡片式布局 | 不像普通聊天框 |
| 准备示例项目入口 | AI试衣、电商退货、小游戏、创业红绿灯自身 | 一键填充示例数据 |

---

### P0-2：项目输入与证据记录

| 任务 | 说明 | 验收 |
|---|---|---|
| 项目输入表单 | 项目名、用户、痛点、替代方案、获客、变现、当前阶段、投入、距上次真实用户行动 | 可以保存并进入下一步 |
| 证据记录表单 | 访谈人数、测试帖、私信、试用、付费、留存等 | 可以计算证据充分度 |
| 本地保存 | 先用 localStorage 或本地状态 | 刷新后尽量不丢数据 |

---

### P0-3：Agent 核心逻辑

| 任务 | 说明 | 验收 |
|---|---|---|
| 假设拆解 | 拆用户、痛点、付费、获客、技术、交付假设 | 输出结构化卡片 |
| 证据充分度评分 | 根据用户行为计算证据分 | 不输出“成功率” |
| 项目结构清晰度 | 判断关键输入是否完整 | 不代表项目可行性，不输出成功率 |
| 交付可控性判断 | 一人可做/砍范围/找合作/暂停 | 必须给路径建议 |
| 红队质询 | 根据证据不足处生成问题 | 至少 3 个角色追问 |
| 灯号判定 | 绿灯/黄灯/红灯/蓝灯 | 有清晰理由 |

---

### P0-4：判定报告

| 任务 | 说明 | 验收 |
|---|---|---|
| 灯号报告 | 当前灯号 + 判断依据 | 用户一眼看懂 |
| 投入上限 | 时间、金钱、禁止事项 | 能防止冲动投入 |
| 7 天验证任务 | 具体到每天/动作 | 不是空泛建议 |
| 停止条件 | 什么情况暂停/放弃 | 有数字化阈值 |
| 复制报告 | 一键复制 Markdown | 可粘贴到文档/聊天 |
| 本轮保存与回填 | 保存灯号和证据状态，回填新行为后再判灯 | 可展示多轮节奏变化 |

---

### P0-5：演示稳定性

| 任务 | 说明 | 验收 |
|---|---|---|
| fallback 示例数据 | 模型失败也能演示 | 现场不崩 |
| 错误提示 | API 失败、空输入、异常数据 | 有友好提示 |
| 演示案例 | 至少 4 个预置案例 | 可快速切换 |
| 自测 | `npm run verify:p0` + `npm run build` | 覆盖四灯、报告结构、localStorage 回退与生产构建 |

## 7. 开发原则

1. 先 P0，后 P1，不要发散。
2. 不做“创业成功率预测器”。
3. 不输出成功率，只输出证据充分度和灯号。
4. UI 要有产品感：灯号、仪表盘、证据卡片、行动卡片。
5. 模型失败也要能演示。
6. API Key 不写死在前端。
7. Codex 先自检，不要频繁要求用户测试。
8. 用户只在关键节点验收：P0闭环、UI效果、部署演示、路演PPT。

## 8. P0 验收目标

P0 已完成代码实现。人工体验验收时，只检查以下闭环，不扩展 P1/P2：

```text
1. 从首页进入项目输入，填写或切换一个演示项目。
2. 在三段六个路测关卡中，补充真实证据与下一步行动方案。
3. 进入“校准报告”，查看灯号、投入上限、7 天任务、停止条件。
4. 点击“保存本轮”，确认历史记录出现新的校准节点。
5. 切换四个演示项目，确认分别能呈现黄灯、红灯、绿灯、蓝灯。

限制：
- 不新增登录、支付、复杂数据库、案例库或联网搜索。
- 当前持久化仅使用 localStorage。
- 当前判定只衡量证据与行动节奏，不输出“创业成功率”。
- 自动自测命令：`npm run verify:p0`；生产构建命令：`npm run build`。
```
